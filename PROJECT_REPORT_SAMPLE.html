1. Project Overview

Journal Trend Analyzer is a Flutter mobile application that helps users explore research topics and inspect publication, journal, keyword, and author trends. OpenAlex is the authoritative publication data source; Firebase provides identity, cloud storage, messaging, analytics, crash reporting, and remote configuration. Four persistent navigation destinations— Home, Journals, Keywords, and Profile— match the Lab 03 assignment specification.

2. Architecture and MVVM Implementation

The project uses an MVVM-compatible layered design organized by feature (Clean Architecture layout). Screens remain focused on rendering and interaction. Riverpod notifiers/providers act as ViewModels, repositories expose domain operations, and services isolate OpenAlex / Firebase SDKs so UI never calls cloud APIs directly.

Topic analysis flow

  1. Normalize the query and record Analytics search_topic.
  2. Load publications, journals and keywords asynchronously.
  3. Compute top authors, influential papers, yearly trend and citation average.
  4. Publish progressive state updates so the dashboard stays responsive.

Reliability decisions

  • Bounded pages and request timeouts protect the mobile client.
  • Future publication years are filtered using the device year.
  • Errors map to user-friendly, retryable UI states (Either<Failure, T>).
  • PDF reports are scoped by Firebase UID under report/{uid}/….

Remote Config & Crashlytics demos

Profile surfaces at least two remote parameters and two crash actions for verification:

  • max_journals_displayed
  • max_keywords_displayed
  • Handled exception → non-fatal recordError
  • Test crash → intentional fatal crash (verification only)

Safety note

“Handled exception” keeps the app running while still sending a diagnostic event. “Test crash” terminates the process and is used only to confirm Crashlytics ingestion after restart. FCM token and notification center are also available on Profile for Messaging verification.

Notification center

Incoming FCM messages are displayed locally (foreground) and stored in the in-app notification center so users can review research alerts without leaving the Profile tab.

Evidence: Firebase Console shortcuts (Authentication, Crashlytics, Messaging, Remote Config, Storage) and Profile UI actions. Storage path pattern: report/{uid}/….

Import directives inconsistently ordered Reduced readability / analyzer consistency Reordered imports in affected screens/providers Fixed

Combined Patrol scenarios collapsed reported cases Evidence did not map 1:1 to assignment Split into eleven independently reported tests Fixed

Challenges encountered

  • OpenAlex rate limits and variable latency required bounded pages, timeouts and retries.
  • Native Google Sign-In / permission dialogs needed Patrol native automation.
  • Remote Config production values differ from local defaults — tests must assert behavior.
  • Crashlytics fatal events upload after process restart.

Lessons learned

  • Layered ViewModel/MVVM code is easier to test and maintain.
  • Cloud evidence must be verified in Console reports, not only DebugView.
  • E2E tests should map directly to assessment scenarios.
  • Dynamic services need robust assertions and user-friendly fallback UI.
